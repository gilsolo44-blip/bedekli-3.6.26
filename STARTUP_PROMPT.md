## 🎯 הזרימה שאני רוצה (GSD-driven)

### שלב 1: Onboarding לפרויקט
תקרא קודם את:
- `CLAUDE.md` — סקירה מלאה של הארכיטקטורה (חובה)
- `research.md` — מחקר באגים מפורט מסשן קודם

### שלב 2: מיפוי הקודבייס
הרץ:
```
/gsd map-codebase
```
זה יספאון 4 mapper agents במקביל וייצר 7 מסמכים מובנים ב-`.planning/codebase/` שיהוו את הזיכרון לטווח ארוך של הפרויקט.

### שלב 3: דיבוג עמוק של הבאגים
הרץ:
```
/gsd forensics
```
או אם זה מהיר מדי:
```
/gsd debug
```
תזהה את שרשרת הכשלים ב-`step1_llm` (שורה 163 ב-`server.js`) ו-`step3_extract` (שורות 702-761).

### שלב 4: תכנון התיקון
הרץ:
```
/gsd plan-phase
```
או לתכנון מעמיק יותר:
```
/gsd ultraplan-phase
```
תייצר תוכנית עבודה אטומית, צעד-צעד, עם בדיקות פר-שלב.

### שלב 5: ביצוע
הרץ:
```
/gsd execute-phase
```
זה ירוץ את התוכנית עם gates ובדיקות אחרי כל שלב.

### שלב 6: בדיקה
הרץ:
```
/gsd code-review
/gsd verify-work
```

### שלב 7: שמירת הידע
הרץ:
```
/gsd extract-learnings
```
זה ישמור את הלמידות ל-`claude-mem` כדי שלסשנים הבאים יהיה זיכרון של ההחלטות הארכיטקטוניות.
