# Archetype-Aware Extraction & Simplification Pipeline
**תאריך:** 2026-06-04  
**סטטוס:** מאושר לתכנון

---

## 1. מטרה

לשפר את דיוק חילוץ הליקויים ב-Bedekli על ידי הזרקת כללי קריאה ספציפיים לפי ארכיטיפ החברה לכל שלב ב-pipeline, ולהוסיף שדה `simplified_explanation` לכל ליקוי — הסבר פשוט לבעל דירה שאינו מהנדס.

### בעיות שנפתרות
| בעיה | שלב אחראי | פתרון |
|------|-----------|--------|
| אין חלוקה לחדרים | `step1_llm` | ארכיטיפ context + ולידציה מינימום 3 סקשנים |
| פרקים → רק כותרות ומספרי עמודים | `step2_filter` + `step1_llm` | זיהוי ודילוג על עמודי תוכן עניינים |
| ליקויים שמפספסים | `step3_extract` | chunking לפי ארכיטיפ + כללי גבול מדויקים |
| מחירים לא מדויקים | `step3_extract` | חילוץ עלויות inline לפי מיקום בארכיטיפ |
| שפה הנדסית לא נגישה | `step3c_simplify` (חדש) | שלב תרגום נפרד לכל ליקוי |

---

## 2. ארכיטקטורה

```
PDF text
   ↓
[step2_filter]      ← שדרוג: סינון TOC pages לפי toc_skip_signals
   ↓
[step1_llm]         ← שדרוג: structure_hint + TOC skip + ולידציה
   ↓
[step2b_byRoom]     ← ללא שינוי
   ↓
[step3_extract]     ← שדרוג מרכזי: ארכיטיפ rules + עלויות inline
   ↓
[step3b_matchCosts] ← ללא שינוי
   ↓
[step3c_simplify]   ← חדש: מוסיף simplified_explanation
   ↓
[step4_schema]      ← ללא שינוי
```

**עיקרון מרכזי:** `data/archetype_rules.json` הוא המקור היחיד לכל כללי הקריאה. כל שלב שואב ממנו את ה-context הרלוונטי לפי הארכיטיפ שזוהה ב-`step0c`.

## 4. הפרומפטים

### 4.1 `step1_llm` — מיפוי מבנה (שדרוג)

```
אתה מנוע מיפוי מבנה לדוחות בדק בית עבריים.

[ארכיטיפ מסמך מזוהה: {{ARCHETYPE}}]
[הנחיית מבנה: {{STRUCTURE_HINT}}]
[סימני תוכן עניינים לדילוג: {{TOC_SKIP_SIGNALS}}]

משימה: זהה את כל הסקשנים (חדרים/פרקים) במסמך.

חוקים קשיחים:
1. דלג על עמודים שמכילים רק כותרות + מספרי עמודים — אלו עמודי תוכן עניינים, לא תוכן אמיתי.
2. אם מצאת פחות מ-3 סקשנים במסמך שמכיל יותר מ-10 עמודים — חפש שוב. סביר שפספסת חלוקה.
3. קבע structureType: "rooms" אם הסקשנים הם חדרים (סלון, מטבח), "chapters" אם הם נושאים (איטום, חשמל).

החזר JSON בלבד, ללא טקסט מסביב:
{
  "structureType": "rooms" | "chapters",
  "sections": [
    { "name": "שם הסקשן", "startPage": N, "endPage": N }
  ]
}

טקסט המסמך:
{{FILTERED_TEXT}}
```

**משתנים מוזרקים מ-archetype_rules.json:**
- `{{ARCHETYPE}}` ← שם הארכיטיפ
- `{{STRUCTURE_HINT}}` ← `rules[archetype].structure_hint`
- `{{TOC_SKIP_SIGNALS}}` ← `rules[archetype].toc_skip_signals.join(", ")`

### 4.2 `step3_extract` — חילוץ ליקויים (שדרוג מרכזי)

```
אתה מנוע חילוץ ליקויים מדוחות בדק בית עבריים.

[ארכיטיפ: {{ARCHETYPE}}]
[סקשן נוכחי: {{SECTION_NAME}}]
[כיצד לזהות גבול ליקוי: {{DELIMITER_PATTERN}}]
[כללי חילוץ לארכיטיפ זה: {{EXTRACTION_HINTS}}]
[היכן מחיר: {{COST_LOCATION}}]

{{FEW_SHOT_BLOCK}}

חלץ את כל הליקויים מהטקסט. אל תשמיט אף ליקוי — עדיף לחלץ יותר מדי מאשר להחמיץ.

כללים קשיחים:
1. desc = מה הבעיה (תיאור הממצא כפי שמופיע בדוח)
2. action = מה צריך לעשות לתיקון — חייב להיות שונה מ-desc. אם הדוח לא מציין — הסק מהקשר.
3. location = {{SECTION_NAME}}, אלא אם כן צוין מיקום ספציפי יותר בטקסט.
4. costRaw = המחיר המדויק כפי שמופיע בטקסט (כולל ₪ וסימני מטבע). אם אין — null.
5. category = אחת מ: CAT-01 ריצוף | CAT-02 טיח/צבע | CAT-03 איטום | CAT-04 רטיבות |
   CAT-05 עובש | CAT-06 בטיחות | CAT-07 נגרות | CAT-08 מטבח | CAT-09 אינסטלציה |
   CAT-10 חשמל | CAT-11 מסגרות/דלתות | CAT-12 חלונות | CAT-13 מרפסות |
   CAT-14 גג | CAT-15 מבנה | CAT-16 כללי

החזר JSON בלבד:
[{
  "location": "...",
  "desc": "...",
  "action": "...",
  "category": "CAT-XX",
  "workType": "repair|replacement|sealing|painting|cleaning|inspection|adjustment|installation|demolish_redo|other",
  "costRaw": "..." | null,
  "simplified_explanation": ""
}]

טקסט לעיבוד:
{{CHUNK_TEXT}}
```

**`{{FEW_SHOT_BLOCK}}` כשיש דוגמה:**
```
דוגמה לקלט ופלט נכון עבור ארכיטיפ זה:
--- קלט ---
{{FEW_SHOT_INPUT}}
--- פלט נכון ---
{{FEW_SHOT_OUTPUT}}
---
```
כשאין דוגמה — `{{FEW_SHOT_BLOCK}}` = מחרוזת ריקה.

### 4.3 `step3c_simplify` — הנגשה (חדש)

```
אתה מומחה להנגשת מידע טכני לקהל הרחב בישראל.

קיבלת רשימת ליקויים מדוח בדק בית הנדסי.
עבור כל ליקוי, מלא את שדה simplified_explanation בהסבר פשוט לבעל דירה.

כללי כתיבה:
1. עד 2 משפטים בלבד.
2. ענה על: מה הבעיה? למה זה חשוב? מה קורה אם לא מטפלים?
3. החלף מונחים טכניים:
   אשפרה → איטום | מישקים → פרזול | קפילריות → ספיגת לחות דרך הקיר |
   קונסטרוקציה → מבנה נושא | ריצוף צף → ריצוף שאינו מחובר לתשתית |
   מיסב → תושבת | כשל קפילרי → חדירת לחות | מישב → מסגרת
4. אל תמציא פרטים שאינם מופיעים ב-desc וב-action.
5. כתוב בגוף שלישי רגיל, לא פנייה ישירה.

קלט — מערך ליקויים עם simplified_explanation ריק:
{{DEFECTS_BATCH}}

החזר את אותו JSON עם simplified_explanation מלא בכל פריט, ללא שינוי שאר השדות.
```

**אופטימיזציה:** `step3c_simplify` מקבל batch של 15 ליקויים בכל קריאה (לא קריאה לכל ליקוי).

## 5. אסטרטגיית Few-Shot

### תהליך בניית דוגמה (5 פעמים — פעם לכל ארכיטיפ)

1. בחר PDF מייצג:

| ארכיטיפ | PDF מומלץ |
|---------|----------|
| HIERARCHICAL | גולדאל הנדסה |
| KW_PARAGRAPH | בדק בית המקורי |
| NUMBERED_FLAT | גלאור מהנדסים |
| TABLE | בדק הבית |
| UNKNOWN | לעד הנדסה |

2. חלץ טקסט גולמי מסקשן אחד (500–800 מילים).
3. כתוב ידנית את ה-JSON הנכון (desc, action, category, costRaw).
4. שמור ב-`archetype_rules.json` תחת `few_shot.input` ו-`few_shot.output`.

### כמות דוגמאות

| מצב | מספר |
|-----|------|
| התחלה | 1 לארכיטיפ |
| אחרי בדיקה — עדיין טועה | הוסף דוגמה שנייה |
| מקסימום יעיל | 3 לארכיטיפ |

## 6. שינויים בקוד (סיכום)

| קובץ | סוג שינוי | תיאור |
|------|-----------|-------|
| `data/archetype_rules.json` | חדש | מילון 5 ארכיטיפים |
| `server.js` → `step2_filter()` | שדרוג קטן | דילוג על TOC pages |
| `server.js` → `step1_llm()` | שדרוג | הזרקת structure_hint + TOC signals + ולידציה |
| `server.js` → `step3_extract()` | שדרוג מרכזי | הזרקת delimiter + hints + few_shot + costRaw inline; chunk size: 3000 tokens עם overlap של 200 tokens בגבולות ארכיטיפ |
| `server.js` → `step3c_simplify()` | חדש | batch simplification, 15 ליקויים בקריאה |
| `data/company_profiles.json` | ללא שינוי | כבר מכיל ארכיטיפים |
